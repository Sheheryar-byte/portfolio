Python's pipe problem — and how I solved it

May 2026 8 min read

It started with a thread I came across on r/Python. Someone asked a question that gets asked at least twice a year in the Python community: will Python ever get a pipe operator?

The replies were familiar — technically sharp, slightly resigned. Yes, the problem is real. No, the core team won't add |>. Here's why, here are your workarounds, good luck.

I didn't close the tab. I started asking a different question: what if the core team is right and the problem is still worth solving?

The problem with nested function calls

In most languages with functional roots — Elixir, F#, Haskell, even JavaScript with the right libraries — you can write data transformations that read the way they execute: left to right, top to bottom, like an assembly line.

Python doesn't have this. When you chain multiple transformations, the standard approach looks like this:

before.py 
# Read this inside-out. Find the data first.
result = list(filter(lambda x: x > 5, map(lambda x: x ** 2, range(10))))

To understand what's happening, you have to start in the middle, work outward, and mentally track the parentheses. Now add two more steps. Now hand it to a colleague. Now come back to it yourself after three weeks.

The best ones are hiding in plain sight — in threads where smart people complain about something and then accept it as unsolvable.

List comprehensions are often suggested as an alternative. They work beautifully for simple cases. But past one or two operations, a comprehension becomes its own puzzle. The structure that was meant to clarify starts to obscure.

Why the core team keeps saying no

This isn't stubbornness — the reasons are genuinely solid. There are three hard problems stacked on top of each other.

1 — Eager evaluation

In languages where |> works well, the right-hand side is lazy. It waits for data. In Elixir, data |> map(func) works because map(func) is a partially applied function holding its breath. Python evaluates eagerly — map(func) would throw immediately because the iterable argument is missing. Making pipes work would require Python to silently rewrite your code at parse time, which violates one of the language's deepest principles.

2 — Auto-currying

To solve eager evaluation you need partial application — functions that accept arguments one at a time and wait. Haskell does this automatically. Python doesn't, and the core team has never wanted it to. Adding currying to support a pipe operator would be a much larger change than the operator itself.

3 — Argument order inconsistency

Even if you solved evaluation and currying, stdlib functions don't agree on where the iterable goes. map(func, data). filter(func, data). But ''.join(data). A pipe operator would have to pick a convention and break everyone's expectations for the cases it doesn't fit.

💡

The core team isn't wrong. But being right about why a native solution is hard doesn't mean the readability problem disappears. Those are two separate questions.

The insight that unlocked everything

The answer was already in the Python ecosystem. I just needed to recognise it.

Pandas. SQLAlchemy. Django's QuerySet. These are some of the most beloved Python libraries ever built, and they all share the same pattern: method chaining on a lazy wrapper object. You call a method, get back an object of the same type, call another method, and so on. The data flows left to right. You read it in the order it executes.

The approach sidesteps every objection that sinks a native pipe operator. No parser changes. No auto-currying. No argument order debates. Just an object that holds your iterable and exposes methods that return new instances of itself.

The question was: why doesn't this pattern exist for plain Python iterables?

Building pipeflow

I built pipeflow — a zero-dependency, fully typed, lazy stream library that brings readable pipelines to any Python iterable.

✕  before — reads inside-out
result = list(
  filter(
    lambda u: u["age"] >= 18,
    filter(
      lambda u: u["active"],
      users
    )
  )
)
✓  after — reads top to bottom
result = (
  Stream(users)
  .filter(lambda u: u["active"])
  .filter(lambda u: u["age"] >= 18)
  .to(list)
)

Every method returns a new Stream wrapping a Python generator — nothing executes until you call a terminal method like .to(list). The full pipeline is built first, then evaluated once in a single pass.

A full real-world example

etl_pipeline.py 
from pipeflow import Stream

result = (
    Stream(raw_users)
    .filter(lambda u: u["active"])
    .filter(lambda u: u["age"] >= 18)
    .map(lambda u: u["name"].upper())
    .sort()
    .to(list)
)
# ["ALICE", "DAVE"]

Pipe operator syntax — for those who want it

pipe_syntax.py 
from pipeflow import pipe, to

# Works on plain lists too — no Stream() wrapper needed
result = (
    range(10)
    | pipe(filter, lambda x: x % 2 == 0)
    | pipe(map, lambda x: x ** 2)
    | to(list)
)
# [0, 4, 16, 36, 64]

The | operator overloading is done via __ror__, which means it works on plain Python lists, generators, and ranges — not just Stream objects.

What's included

  • .map(func)
  • .filter(func)
  • .flatmap(func)
  • .batch(n)
  • .unique(key)
  • .sort(key)
  • .take(n) / .skip(n)
  • .group_by(key)
  • .tap(func)
  • .fold(func, init)
  • .first() / .last()
  • .to(collector)

Every method is lazy. Nothing evaluates until you call a terminal like .to(list), .count(), or .for_each(). The whole library has zero runtime dependencies and ships with complete type annotations.

What I learned from this

The most useful open source tools rarely come from inventing something new. They come from noticing a gap between a real, recurring frustration and the solutions people have accepted as "good enough."

The pipe operator debate has been running for years. The community had settled on this is a language limitation, live with it. But the readability problem and the question of whether Python should add |> are two completely separate things. You can answer the second question with "no" and still solve the first one.

That reframe — separating the language design question from the user experience question — is where pipeflow came from. The best problems to solve are often hiding inside arguments that have already been won.
